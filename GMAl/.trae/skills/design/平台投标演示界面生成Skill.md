# 平台投标演示界面生成 Skill

## 1. Skill 目标

用于根据用户提供的业务需求、指标说明、参考截图、已有 HTML 或功能描述，生成一套 **可运行、可演示、可交付的投标级前端界面**。

生成结果应满足：

- 适合投标、汇报、售前演示、产品方案展示。
- 页面视觉统一，具备专业后台系统质感。
- 支持真实可点击、可筛选、可联动、可下钻的演示交互。
- 数据应正式、合理、贴近政务、医疗、数据治理、平台建设场景。
- 输出应为离线可运行 HTML 或 ZIP 包，不依赖外部 CDN。
- 若用户是在已有版本上调整，应 **只改指定内容，保留其他区域不变**。

## 3. 总体设计原则

### 3.1 投标演示优先

界面不是普通后台页面，而是用于投标演示，因此要做到：

- 第一眼有产品感。
- 信息结构清晰。
- 业务逻辑可信。
- 点击后有真实反馈。
- 关键流程能串起来演示。
- 不做空壳页面，不做只有静态截图的假界面。

### 3.2 专业、克制、清晰

整体风格应避免过度花哨。

推荐风格：

- 深蓝、科技蓝、灰蓝、白底专业后台风格。
- 小圆角、细边框、轻阴影。
- 表格、筛选区、卡片、弹窗、抽屉风格统一。
- 页面要紧凑，但不能拥挤。
- 不出现大面积无意义留白。
- 不出现文字溢出、错位、重叠、按钮不齐等问题。

### 3.3 真实业务逻辑

数据、字段、流程要符合真实平台建设逻辑。

例如：

- 主数据平台要体现数据接入、标准化、映射、发布、订阅、推送。
- 指标平台要体现原子指标、派生指标、复合指标、计算口径、SQL 生成、规则配置。
- 大屏配置平台要体现组件、画布、指标匹配、预览、发布。
- 转诊大屏要体现上转、下转、机构流向、诊断排名、患者明细。

不得随意堆砌无关模块。

## 4. 视觉规范

### 4.1 整体风格

优先采用以下风格：

- 科技感后台系统。
- 蓝色主色调。
- 干净、专业、投标级。
- 小圆角。
- 弱发光。
- 细线分割。
- 信息密度适中。

避免：

- 圆角过大，显得卡通化。
- 大面积渐变堆叠。
- 模块边框太粗。
- 字体过大或过小。
- 背景太花影响阅读。
- 所有组件长得完全一样。

后续所有后台系统界面统一使用较小、精致的圆角。


避免使用 16px、20px 以上的大圆角，除非用户明确要求。

推荐：

```css

  "Microsoft YaHei",
  "PingFang SC",
  "Helvetica Neue",
  Arial,
  sans-serif;
```

字号建议：

```text

模块标题：15px - 18px
正文：13px - 14px
表格：12px - 13px
辅助说明：12px
```

要求：

- 字体必须清晰。
- 大屏文字不能太小。
- 表格文字不能溢出。
- 按钮文字要垂直居中。
- 中文字段不能乱码。
- 不使用不稳定在线字体。

## 5. 布局规范

### 5.1 PC 管理端布局

常用结构：

```text
顶部导航 / 页面标题区
左侧菜单或业务导航
主内容区
  - 查询筛选区
  - 指标概览卡片
  - Tab / 分类切换
  - 数据表格 / 图表 / 配置区
  - 分页 / 操作区
抽屉 / 弹窗 / 详情页
```

要求：

- 查询区高度适中。
- 查询、重置、新增、导入、导出按钮对齐。
- 筛选项不能重叠。
- 表格列宽合理。
- 操作列不换行。
- 操作按钮数量过多时使用“更多”。
- 重要操作按钮放右上方。
- 页面底部不要出现大面积空白。

### 5.2 大屏布局

常用 16:9 结构：

```text
顶部：标题 + 时间 + 单位/区域选择
左侧：指标卡片 + 图表
中间：地图 / 3D 模型 / 核心态势
右侧：排名 / 趋势 / 结构分析
底部：明细表格 / 流动监测 / 事件列表
```

要求：

- 中间地图或模型要突出。
- 左右组件数量均衡。
- 地图与两侧组件之间可做柔和光效过渡。
- 数据飞线、流动线要有方向感。
- 大屏上所有图表要服务业务，不要纯装饰。
- 标题清晰，不能出现字体重叠。

### 5.3 画布配置类布局

适用于大屏生成平台、场景配置、组件编排。

推荐结构：

```text
顶部工具栏：
  返回、保存、预览、发布、导入、下载指标清单

左侧：
  组件库 / 场景模板 / 数据源 / 指标库

中间：
  可视化画布区域
  支持 PC / 大屏模式切换

右侧：
  页面属性 / 组件属性 / 数据绑定 / 样式配置 / 交互配置

底部或弹窗：
  指标匹配结果
  人工确认
  自动生成记录
```

要求：

- 左中右结构清楚。
- 画布区域视觉突出。
- 属性配置不能过于空泛。
- 要体现“自然语言描述生成画布”或“上传指标生成画布”的能力。
- 生成后支持人工确认、手动匹配、编辑调整。

## 6. 交互规范

### 6.1 必须具备真实演示交互

生成界面时，应尽量提供以下交互能力：

- Tab 切换
- 左右联动
- 筛选查询
- 重置
- 新增
- 编辑
- 删除
- 复制
- 预览
- 发布
- 导入
- 导出
- 下钻明细
- 分页
- 抽屉配置
- 弹窗确认
- 状态切换
- 数据刷新
- 图表联动
- 表格行点击

不能只做静态页面。

### 6.2 数据联动要求

示例：

- 点击某个机构，右侧图表和明细同步变化。
- 切换“全部 / 上转 / 下转”，趋势图、排名、患者明细同步变化。
- 选择某个字典分类，右侧字典项和映射关系同步变化。
- 点击指标卡片，下钻表格数量应与卡片统计一致。
- 切换映射方向，表头和主键字段应动态调整。
- 切换时间粒度，趋势图数据要随之变化。

### 6.3 抽屉与弹窗规范

新增、编辑、配置类操作优先使用右侧抽屉。

抽屉内容应包含：

- 基础信息
- 业务配置
- 数据配置
- 规则配置
- 通知配置
- 保存 / 关闭按钮

要求：

- 新增和维护尽量复用同一抽屉。
- 关闭按钮旁边应有保存按钮。
- 表单项分组清晰。
- 必填项、状态、说明要完整。
- 不要出现无效按钮。

## 7. 数据规范

### 7.1 数据要正式

避免使用：

```text
测试数据1
表1
门急诊就诊记录宽表2
张三
李四
机构A
系统B
示例字段
```

推荐使用更正式的虚拟命名：

```text
东海市人民医院
东海市第二人民医院
城东社区卫生服务中心
北湾镇卫生院
区域全民健康信息平台
基层医疗卫生系统
医院 HIS 系统
电子病历系统
检验信息系统
影像归档系统
医保结算系统
```

### 7.2 避免真实敏感信息

投标演示中应使用虚拟地区、虚拟机构、虚拟人员。

如果用户要求使用真实地区，例如青岛、栾川、威信县，可以使用地区名称，但机构和人员可尽量虚拟化，避免真实个人信息。

### 7.3 数据关系必须自洽

要求：

- 总数 = 各分类之和。
- 卡片统计 = 下钻明细总数。
- 排名数据与表格数据一致。
- 筛选后的数据要合理变化。
- 百分比不能超过 100%。
- 环比、同比、趋势数据要符合常识。
- 明细数据数量要支持分页。

### 7.4 医疗业务数据规范

常见字段：

```text
机构名称
业务日期
门急诊人次
住院人次
出院人次
医疗收入
药品收入
检查检验收入
医保费用
转诊类型
转诊机构
接收机构
诊断名称
患者类型
科室名称
医生姓名
指标编码
指标名称
指标口径
统计周期
数据来源
```

避免不合理指标混搭。

## 8. 指标平台专用规范

### 8.1 原子指标

原子指标是最基础、不可再拆分的医疗行为计数或金额类统计。

页面配置应包括：

- 指标基础信息
- 指标编码
- 指标名称
- 业务域
- 指标口径
- 数据源
- 数据表映射
- 字段映射
- 过滤条件
- 统计方式
- 生成 SQL
- 调度周期

示例：

```text
原子指标：门诊人次
数据源：HIS 门诊就诊视图
统计字段：VISIT_ID
统计方式：COUNT(DISTINCT VISIT_ID)
过滤条件：VISIT_DATE = {业务日期}
```

### 8.2 派生指标

派生指标是在原子指标基础上叠加时间、机构、科室、医生、患者类型等维度形成的指标。

示例：

```text
日门诊量 = 门诊人次 + 日期维度
呼吸科门诊量 = 门诊人次 + 科室维度
儿科月入院人数 = 入院人次 + 月份 + 儿科
```

### 8.3 复合指标

复合指标必须体现：

> 由原子指标和派生指标通过系统配置计算关系形成。

页面中必须明确出现：

```text
原子指标
派生指标
计算关系
公式配置
权重配置
条件评分
系统自动生成计算任务
```

示例：

```text
次均门诊费用 = 门诊总费用 / 门诊人次
床位使用率 = 实际占用总床日数 / 开放总床日数 × 100%
处方合格率 = 合格处方数 / 抽查处方总数 × 100%
```

### 8.4 指标异常规则

规则管理应包括：

- 阈值规则监测
- 模型智能监测
- 同比阈值
- 环比阈值
- 上限阈值
- 下限阈值
- 监测频率
- 通知方式
- 适用机构
- 状态
- 操作

阈值配置要求：

- 上限阈值、下限阈值、同比阈值、环比阈值都应支持大小关系选择。
- 支持数值录入。
- 例如：大于、小于、大于等于、小于等于、介于、不等于。
- 自定义监测频率点击后再展开详细周期和执行时间。
- 系统消息需配置角色。
- 短信需填写手机号。
- 邮件需填写邮箱。
- 已配置人员应在下方列表展示，并支持编辑、移除。

## 9. 主数据和值域映射规范

### 9.1 主数据概览

应体现：

- 主数据总量
- 今日新增
- 今日更新
- 待审核
- 已发布
- 订阅数量
- 推送次数
- 异常数量
- 机构分布
- 系统分布
- 数据趋势
- 推送分析

要求：

- 更新和审核数据列分开。
- 全部数量要等于各机构之和。
- 下钻明细数量要一致。
- 筛选项要符合主数据场景，不要乱加业务域或无关系统。

### 9.2 平台字典

平台字典应作为独立菜单或一级功能保留。

内容包括：

- 字典分类
- 字典项
- 启用项数
- 停用项数
- 待确认数
- 来源系统
- 维护状态
- 更新时间

左侧展示各机构、各系统上送的字典分类。

右侧展示当前字典下的字典项和维护信息。

### 9.3 值域映射

值域映射应支持：

- 按机构筛选
- 按接入系统筛选
- 按字典分类筛选
- 映射方向切换
- 自动匹配
- 手动确认
- 批量审核
- 导入导出
- 映射状态统计

映射方向包括：

```text
接入系统 → 平台
平台 → 接入系统
```

切换方向时：

- 表头要动态变化。
- 主方向字段作为主键。
- 对方映射字段作为匹配结果。
- 不允许表头和数据不匹配。

## 10. 大屏设计专用规范

### 10.1 大屏比例

默认使用：

```text
16:9
1920 × 1080
```

如果用户要求 4K，应按：

```text
3840 × 2160
```

如果用户要求竖屏，可使用：

```text
9:16
```

### 10.2 青岛转诊大屏示例规范

如果生成“青岛转诊大屏”，应包含：

- 单位时间选择
- 转诊总人数
- 上转总人数
- 下转总人数
- 转诊人次趋势
- 患者基本情况
- 性别结构
- 年龄结构
- 转诊明细
- 转诊机构数量排名
- 转诊诊断排名
- 中间青岛地图
- 飞机线 / 光效飞线
- 转入和转出的流动特效

交互要求：

- 全部、上转、下转可切换。
- 切换后卡片、趋势、排名、明细同步变化。
- 地图飞线方向随上转/下转变化。
- 明细表格自动滚动或分页。
- 地图不能只是背景，要体现流向关系。

### 10.3 地图与飞线

地图应满足：

- 区域轮廓清晰。
- 中心区域突出。
- 机构点位清楚。
- 飞线有方向。
- 转入和转出用不同视觉区分。
- 可加入光点流动动画。
- 不要遮挡核心数据。

## 11. 画布生成平台规范

针对“自然语言生成大屏 / 上传指标生成画布 / AI 生成平台”类需求，应包含以下模块。

### 11.1 场景管理

原“大屏列表”应升级为“场景管理”。

功能包括：

- 场景分类
- 场景列表
- 预览
- 复制
- 编辑
- 新增
- 导入
- 重命名
- 删除
- 发布状态
- 最近更新时间
- 创建人
- 应用范围

页面设计应参考专业后台卡片 + 表格组合。

### 11.2 画布配置

点击新增或编辑后进入画布配置界面。

需要支持：

- PC 画布
- 大屏画布
- 模板选择
- 组件拖拽
- 图层管理
- 组件属性配置
- 数据绑定
- 指标绑定
- 交互配置
- 样式配置
- 预览
- 保存
- 发布

### 11.3 自然语言生成

应提供输入区：

```text
请输入大屏或报表需求描述
```

示例能力：

- 根据描述识别业务主题。
- 自动推荐页面布局。
- 自动匹配指标。
- 自动选择组件。
- 自动生成画布。
- 支持人工确认。
- 支持生成后继续编辑。

### 11.4 指标上传生成

支持上传：

- Word
- Excel
- Markdown

上传后流程：

```text
文件解析
指标识别
指标标准化
匹配数仓指标库
人工确认
生成组件布局
生成画布
保存 / 预览 / 发布
```

### 11.5 指标清单下载

发布或预览前，应支持按照画布模块分布下载指标清单。

指标清单包括：

- 页面名称
- 模块名称
- 组件名称
- 指标编码
- 指标名称
- 指标口径

## 12. 前端实现规范

### 12.1 输出形式

优先输出：

```text
单个 HTML 文件
```

或：

```text
离线 ZIP 包
  index.html
  css/
  js/
  data/
  assets/
```

如果用户要求“png 融入 html”，则应将图片资源内嵌到 HTML 中，避免单独文件夹依赖。

### 12.2 离线可运行

要求：

- 不依赖 CDN。
- 不引用外部字体。
- 不引用在线 JS。
- 不引用在线 CSS。
- 图表库如需使用，应本地化或使用原生 SVG / Canvas / CSS 模拟。
- 打开 `index.html` 即可运行。

### 12.3 代码要求

代码应：

- 结构清晰。
- 注释适度。
- CSS 统一管理。
- JS 交互可读。
- 数据集中定义。
- 不写死过多重复 DOM。
- 避免控制台报错。
- 避免横向滚动条。
- 页面适配主流浏览器。

### 12.4 图表实现

可使用：

- 原生 SVG
- Canvas
- CSS 图表
- 本地化 ECharts

图表类型应根据业务选择：

- 趋势：折线图
- 排名：横向柱状图
- 结构：环形图 / 饼图
- 分布：柱状图 / 堆叠图
- 流向：地图飞线 / 桑基风格
- 明细：表格

不要所有图表都用环形图。

## 14. 修改与迭代规范

当用户基于已有 HTML 或截图提出修改时，必须遵守以下规范。

### 14.1 只改指定内容

如果用户说：

```text
其他地方不要做改动
```

则必须严格只调整指定区域，不重构页面，不改变整体样式，不改无关模块。

### 14.2 保留已确认设计

对于用户已经满意的内容：

- 不随意调整布局。
- 不更换配色。
- 不修改菜单。
- 不改圆角。
- 不改变按钮位置。
- 不删除已有功能。

### 14.3 增量修改优先

除非用户明确要求重做，否则：

```text
在当前版本基础上增量修改
```

不要重新生成完全不同的一版。

### 14.4 必须关注截图反馈

用户提供截图指出问题时，应重点解决：

- 对齐问题
- 宽度问题
- 字体重叠
- 表格溢出
- 菜单缺失
- 切换无效
- 按钮错位
- 数据不一致
- 视觉不统一

## 15. 质量验收标准

生成完成后，界面应满足以下检查项。

### 15.1 视觉检查

- 页面风格统一。
- 蓝色科技感或后台专业感明确。
- 组件圆角符合小圆角规范。
- 表格、按钮、输入框对齐。
- 无明显空白。
- 无文字溢出。
- 无重叠。
- 无乱码。
- 字体清晰。

### 15.2 交互检查

- Tab 可切换。
- 查询重置有效。
- 下拉筛选有效。
- 左右联动有效。
- 新增编辑弹窗或抽屉可打开。
- 保存关闭逻辑完整。
- 表格分页有效。
- 图表随筛选变化。
- 预览、发布等按钮有反馈。

### 15.3 数据检查

- 总数与明细一致。
- 卡片数据与表格数量一致。
- 排名数据合理。
- 百分比合理。
- 趋势数据合理。
- 业务字段正式。
- 无测试占位数据。

### 15.4 交付检查

- HTML 可直接打开。
- 离线可运行。
- 无 CDN 依赖。
- 无控制台明显报错。
- 文件结构清晰。
- ZIP 包可解压运行。
- 图片资源如有需要，应内嵌或本地化。
