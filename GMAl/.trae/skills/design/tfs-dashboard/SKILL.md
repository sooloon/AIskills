# TFS 需求分析看板生成器

## When to use

- 用户需要生成或重新设计 TFS 需求分析看板
- 用户提到 "TFS看板"、"需求分析看板"、"生成看板"、"需求分析dashboard"
- 用户需要包含组织架构、KPI、流程状态和排名分析的可视化报告页面

## When NOT to use

- 用户需要连接实时数据库的 BI 看板 → 需要数据对接方案
- 用户只需要简单图表而非完整的 TFS 需求分析体系
- 用户需要非 HTML 格式的看板（如 PPT、PDF 报告）

## Inputs

- 三级部门架构数据（一级/二级/三级部门名称和关系）
- 需求数据（创建/关闭时间、状态、处理人、部门归属）
- 输出路径：默认 `<工作目录>/需求分析看板.html`

## Procedure

### Step 1: 构建页面框架

创建自包含 HTML 文件，包含以下区域：

1. **头部区域**：蓝色渐变导航栏 + 全局筛选区（三级部门架构 + 时间选择）
2. **KPI 概览**：8 个核心指标卡片（需求积压、本月创建/关闭、平均时长、按时率、需求变更、任务积压、工作量）
3. **需求流程状态**：甘特图式流程展示 已创建→需求分析→开发中→测试中→已关闭
4. **部门架构分布**：树状图（三级部门）+ 环形图（二级部门占比）
5. **三级部门排名**：积压排名 TOP8（横向条形图）+ 效率排名 TOP8（按时率+sparkline）
6. **趋势与效率对比**：创建/关闭趋势图（分组柱状图）+ 二级部门能力雷达图（6 维）
7. **人员明细表**：姓名、部门、角色、积压、已完成、按时率、平均时长、本周新增、状态

### Step 2: 应用设计规范

```css
/* 布局 */
12 列网格系统，响应式断点：1400px / 1024px

/* 颜色 */
主色: #1890ff (蓝)
成功: #52c41a (绿)
警告: #fa8c16 (橙)
错误: #f5222d (红)
紫色: #722ed1
青色: #13c2c2
```

图表类型：甘特图、树状图、环形图、横向条形图、分组柱状图、雷达图、Sparkline

### Step 3: 填充组织架构数据

**一级部门**：区域卫生研发中心

**二级部门（5 个）**：智慧公卫研发部、基层医疗研发部、区域卫生产品部、区域应用研发部、区域平台研发部

**三级部门（18 个）**：按二级部门分组，详见各研发部下属团队。

### Step 4: 输出验证

生成后检查：
- [ ] 全局筛选是否影响排名展示
- [ ] 页面是否有大量留白
- [ ] 图表类型是否多样化（≥5 种图表类型）
- [ ] 需求状态是否用甘特图表示流程关系
- [ ] 雷达图题注是否显示完整

## Output Format

输出文件：`<工作目录>/需求分析看板.html`

文件特点：完全自包含、内嵌 CSS、含示例数据、响应式设计、无外部依赖。

## Examples

### Example 1: 生成看板

**用户输入**: 生成 TFS 需求分析看板

**处理流程**:
1. 按 Procedure Step 1-3 构建完整 HTML
2. 填充示例数据
3. 输出文件路径

**输出**:

### Example 2: 重新设计

**用户输入**: 重新设计需求分析看板，优化布局

**处理流程**:
1. 读取现有看板文件
2. 分析现有布局问题（留白、图表类型单一等）
3. 按 Procedure 重新生成

## Error Handling

- 输出目录无写入权限 → 提示用户检查目录权限或指定其他路径
- HTML 文件已存在 → 询问用户是否覆盖，或自动备份为 `.bak`
- 组织架构数据不完整 → 使用默认示例数据并提示用户补充

## ⚠️ Gotchas

- **不要硬编码绝对路径** — 使用工作目录的相对路径，方便在不同环境使用
- **不要遗漏甘特图** — 需求流程状态必须用甘特图表示，这是核心区分点
- **图表类型不要少于 5 种** — 纯数字表格不是看板，需要多样化可视化
- **不要引入外部 CDN 依赖** — 文件必须完全自包含，方便离线使用和内网部署
- **不要忽略响应式设计** — 需在 1400px 和 1024px 两种宽度下测试布局
- **雷达图维度必须对称** — 6 个维度值需归一化到相同量级，否则某一维会撑爆图表
